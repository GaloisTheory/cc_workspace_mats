The infra/vast harness — how it works, and how hard a move to Modal would be

A walkthrough for someone who has never run jobs on rented cloud GPUs before. Part 1 explains every moving piece of the harness you vibe-coded. Part 2 is an honest assessment of porting it to Modal (the H100-availability question).

1 · The mental model: you are renting a stranger's computer by the hour

If you've only ever used a persistent box (SSH in, the files are still there tomorrow), the key shift is this:

Your old GPU box

  • One machine, always yours
  • Disk persists between sessions
  • You install things once
  • You SSH in and run tmux by hand
  • Idle = still yours, just not billed for compute

Vast.ai

  • A marketplace of other people's idle GPUs
  • You search offers, pick one, rent it per hour
  • The machine is ephemeral: destroy it and everything on its disk is gone
  • It can be preempted (bid/spot instances especially)
  • Billed while it exists — even stopped, storage bills

Because the machine is disposable, the entire harness is designed around one principle:

The rented box holds nothing of value permanently. The Docker image defines the environment, Git holds the code, and Hugging Face / W&B hold the data, models and results. The instance is a throwaway worker that pulls all three in, does the work, pushes results out, and is then deleted.

2 · The five pieces

Everything lives in infra/vast/ plus one Claude skill. Each file has one job. The colour tag says where it runs: your laptop baked into the image on the rented box.

FileRunsWhat it does
Dockerfile image Defines the environment: CUDA 12.8, Python 3.10, uv, git, tmux. Crucially it does not copy your repo or install project deps — it stays generic so one image works for every project. It bakes run_job.sh in as /usr/local/bin/vast-run-job and makes it the container's default command.
search_offers.sh laptop A thin wrapper over vastai search offers. You give it minimum GPU count, GPU RAM, reliability; it lists machines you could rent and their hourly price. You read the list and pick an OFFER_ID.
create_instance.sh laptop The launcher. Takes your OFFER_ID, IMAGE, TRAIN_CMD (+ optional repo, ref, setup cmd, secrets), packages them up, and tells Vast: "rent that machine, run that image, and on startup run vast-run-job."
run_job.sh rented box The actual job runner, executed inside the container on the rented machine. Unpacks the instructions, loads secrets, logs into HF/W&B, clones the repo at the right commit, runs setup, then training, then post-training — logging everything.
.claude/skills/vast-run/SKILL.md laptop Not code — an operator playbook for Claude. The hands-on alternative: SSH in, sync code, launch under tmux, monitor GPUs, checkpoint to HF, recover from preemption. Used when you want eyes on the run instead of fire-and-forget.

3 · There are actually two ways to run a job

This is the part that's easy to miss. The harness supports two distinct workflows that share the same image but otherwise barely overlap.

A · Scripted & headless

search_offers.sh → create_instance.sh → the container auto-runs vast-run-job on boot via Vast's --onstart-cmd.

Fire-and-forget. You don't SSH in. The job starts itself the moment the container boots. Good for "I know exactly the command, just run it."

B · Interactive operator (the skill)

The /vast-run skill: search offers, create a bare instance, SSH in, rsync/clone code, launch under tmux, watch nvidia-smi, push checkpoints, recover.

Hands-on. Survives SSH drops because the job lives in remote tmux. Good for long, expensive, or fragile runs you want to babysit.

These two paths are not unified. Path A's run_job.sh does no monitoring and no checkpoint upload. Path B's monitoring/recovery lives only in the skill prose, not in any script. Knowing which path you're on explains most "why didn't it do X?" confusion.

4 · End-to-end data flow (the scripted path)

1
Build & push the image laptop — docker build -f infra/vast/Dockerfile then docker push to a registry like ghcr.io/<you>/cc-workspace-vast:cuda128-uv. Done once; reused across runs.
2
Find a machine laptop — MIN_GPUS=8 infra/vast/search_offers.sh lists rentable offers. You pick an OFFER_ID.
3
Launch laptop — set env vars (IMAGE, GIT_REPO, GIT_REF, PROJECT_DIR, SETUP_CMD, TRAIN_CMD, secrets) and run create_instance.sh. It base64-encodes every command string and passes them as Docker env vars, then calls vastai create instance … --onstart-cmd "vast-run-job".
4
Vast provisions — finds the host, pulls your image, starts the container. tini is PID 1; it launches vast-run-job (which is run_job.sh).
5
The box bootstraps itself rented box — run_job.sh base64-decodes the commands, sources /workspace/.secrets if present, logs into HF & W&B, git clones the repo, checks out GIT_REF, inits submodules, runs SETUP_CMD (e.g. uv sync --frozen).
6
It trains rented box — runs PRETRAIN_CMD → nvidia-smi → TRAIN_CMD → POSTTRAIN_CMD, teeing each into /workspace/logs/$RUN_NAME/.
7
Results must escape — the harness does not upload anything. Whatever you want to keep, your TRAIN_CMD / POSTTRAIN_CMD must push to Hugging Face or W&B itself.
8
You destroy the instance laptop — vastai destroy instance <id>. Manual. Forget this and it bills indefinitely.
[laptop] build+push image ──────────────► ghcr.io registry [laptop] search_offers.sh ──► pick OFFER_ID [laptop] create_instance.sh │ base64(TRAIN_CMD, SETUP_CMD, GIT_REF…) + secrets as env ▼ vastai create instance --onstart-cmd vast-run-job │ ▼ [rented box] tini → run_job.sh ├─ decode env, source /workspace/.secrets ├─ huggingface-cli login / wandb login ├─ git clone <GIT_REPO> @ <GIT_REF>, submodules ├─ SETUP_CMD (uv sync --frozen, flash-attn…) ├─ PRETRAIN_CMD ; nvidia-smi ├─ TRAIN_CMD ──► results pushed to HF / W&B (by YOUR cmd) └─ POSTTRAIN_CMD [laptop] vastai destroy instance <id> ← manual

5 · Why it's built this way (the non-obvious decisions)

Why base64-encode the commands?

Your TRAIN_CMD is a full shell line with spaces, quotes, ${GPU_COUNT:-8}, etc. It has to survive being passed through Docker's -e VAR=value and Vast's own env plumbing. Quoting that safely through two layers is a nightmare; base64 turns it into one boring ASCII token. run_job.sh decodes it back on the other side. This is the single cleverest bit of the harness.

Why doesn't the image contain the repo or dependencies?

So one image serves every project. The repo is cloned at run time at a specific GIT_REF, and deps are installed by SETUP_CMD. Trade-off: every run re-installs dependencies (slow, e.g. flash-attn compiles each time). The README explicitly suggests a project-specific derived image later if that solve gets painful.

Why are secrets passed at launch, never baked in?

A Docker image pushed to a registry is effectively public to anyone who can pull it. Baking HF_TOKEN in would leak it. Instead secrets are injected as runtime env (or read from /workspace/.secrets on the box) and the launcher only forwards the ones that exist.

Why is Hugging Face the "source of truth"?

Because the rented disk is not portable. Vast volumes can warm-cache one specific host, but they don't move between hosts and they vanish on destroy. So models, datasets, checkpoints and final results all have to live in HF / W&B / S3 — the box only ever holds a working copy.

6 · Rough edges (this was vibe-coded — here's what to actually watch)

No automatic artifact upload. run_job.sh only tees logs to the box's local disk. If the instance dies mid-run, logs and checkpoints die with it unless your TRAIN_CMD itself streams them out. The /vast-run skill compensates with manual HF uploads — but only on Path B.
No automatic teardown. Nothing destroys the instance. A forgotten box bills 24/7, and a stopped box still bills for storage. This is the most expensive failure mode.
Scripted path is blind. Path A has zero monitoring — no heartbeat, no "is it still alive", no ETA. You either trust it or SSH in manually (which is really Path B).
Preemption is unhandled by the harness. Bid/spot instances can be killed at any moment. There's no checkpoint/resume logic in run_job.sh; that responsibility is silently pushed onto your training script.
Private-repo cloning is not really solved on Path A. run_job.sh just does git clone $GIT_REPO. For an git@github.com:… URL the container needs an SSH deploy key that nothing injects. In practice you need an HTTPS token in the URL or a key, which the scripted path doesn't set up for you.
Image hygiene is manual discipline. The README says "use a pinned tag, not latest" and "rebuild when the Dockerfile changes" — but nothing enforces either. Run the wrong stale tag and you'll debug a phantom.

7 · Moving to Modal — the big picture

Your motivation is concrete: Vast doesn't reliably have 8×H100 available, and you're price-insensitive. Modal is a serverless GPU platform with deep H100 capacity. The good news is that the conceptual model is simpler than Vast, and most of the harness's complexity exists only to paper over things Modal does natively.

Vast model

A marketplace. You shop for a physical machine, rent it, SSH in or auto-run a command, and are responsible for its entire lifecycle (find, launch, monitor, upload, destroy). The harness is mostly lifecycle glue.

Modal model

Serverless. You write a Python function decorated with the GPU it needs; modal run provisions, runs, streams logs, and tears down automatically. No marketplace, no SSH, no offer IDs, no manual destroy, no base64 trick.

Verified facts for your case (Modal docs, May 2026):

  • 8×H100 on one container is supported: @app.function(gpu="H100:8"). Up to 8 GPUs / 1,536 GB on one machine, always physically co-located (so accelerate multi-GPU works unchanged).
  • H100s are SXM and may auto-upgrade to H200 free; pin with gpu="H100!" if you need exactly H100.
  • Max function timeout is 24 h. Multi-day runs must checkpoint and resume across invocations (you should be doing this on Vast anyway, since spot preemption has the same requirement).
  • Detached runs exist (modal run --detach / .spawn()) so the job survives your laptop closing — the equivalent of remote tmux, for free.
  • You can reuse your existing Dockerfile almost verbatim via modal.Image.from_dockerfile(...) — the environment does not need rethinking.

9 · What the Modal version actually looks like

The entire scripted path collapses into roughly this — one file replaces create_instance.sh + run_job.sh + the onstart wiring:

# modal_app.py  (illustrative — replaces create_instance.sh + run_job.sh)
import subprocess, modal

image = (
    modal.Image.from_dockerfile("infra/vast/Dockerfile")   # reuse what you have
    .run_commands("git config --global --add safe.directory '*'")
)

app = modal.App("dare-train")

@app.function(
    gpu="H100:8",                       # <- the whole point: reliable 8xH100
    image=image,
    timeout=24 * 60 * 60,               # 24h max; checkpoint+resume beyond that
    secrets=[modal.Secret.from_name("hf-wandb")],   # HF_TOKEN, WANDB_API_KEY
    volumes={"/workspace/.cache": modal.Volume.from_name("hf-cache", create_if_missing=True)},
)
def train(git_ref: str, setup_cmd: str, train_cmd: str):
    sh = lambda c: subprocess.run(["bash", "-lc", c], check=True)
    sh(f"git clone https://github.com/<org>/<repo>.git /workspace/repo "
       f"&& cd /workspace/repo && git checkout {git_ref} "
       f"&& git submodule update --init --recursive")
    sh(f"cd /workspace/repo/projects/dare && {setup_cmd}")
    sh(f"cd /workspace/repo/projects/dare && {train_cmd}")   # results -> HF/W&B

@app.local_entrypoint()
def main():
    train.remote(
        git_ref="migration/gpu-transfer",
        setup_cmd="uv sync --frozen && MAX_JOBS=16 uv pip install flash-attn --no-build-isolation",
        train_cmd="uv run accelerate launch --num_processes 8 experiments/train/train_lora.py --split 1",
    )

Launch is then just modal run modal_app.py (add --detach for laptop-can-close). Note how the git clone → checkout → submodules → SETUP_CMD → TRAIN_CMD spine is literally the same logic as run_job.sh — that's why the port is mechanical, not a redesign.

10 · Difficulty verdict

Overall: low-to-moderate, and the result is simpler than the Vast harness. ~1–2 focused days to a working DARE run

  • What's genuinely portable (≈80% of the value): the training/eval commands, the clone→checkout→setup→train spine, HF/W&B as source of truth, and the Dockerfile. None of that is Vast-specific — it carries over essentially unchanged.
  • What you rewrite: one Python file (modal_app.py). It replaces three Vast files and is shorter than any one of them, because the base64 trick, offer search, onstart wiring, and manual teardown all simply cease to exist.
  • What gets better for free: reliable H100 capacity (your actual goal), auto teardown (kills the worst cost risk), streamed logs, native retries/timeouts, managed secrets, optional persistent cache volume.
  • What you give up / new constraints: the 24 h timeout forces explicit checkpoint+resume for multi-day runs (you arguably need this on Vast spot anyway); no raw SSH — interactive debugging becomes modal shell; you adopt Modal's Python-first idioms and a soft vendor lock-in (acceptable given you're price-insensitive and the entrypoint is thin).

The reason it's not a big lift: most of the Vast harness is glue that exists only because you own the machine's lifecycle. Modal owns that lifecycle for you, so a large chunk of the code you wrote doesn't get ported — it gets deleted. The risk isn't the migration mechanics; it's verifying multi-GPU accelerate behaviour and checkpoint/resume across the 24 h boundary on a representative run.
